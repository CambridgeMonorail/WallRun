# Getting Started

This page is the shortest path from an empty file to a credible signage screen.

## Run The Workspace

```bash
git clone https://github.com/CambridgeMonorail/WallRun.git
cd WallRun
pnpm install
pnpm storybook
```

Storybook runs at `http://localhost:4400`.

If you want the demo website as well:

```bash
pnpm serve:client
```

## Install Components Into An Existing App

If you already have a React app and only want the signage components, use the published shadcn registry instead of cloning the whole monorepo.

Before running these commands, make sure you are inside a React app that is already set up for Tailwind CSS and `shadcn/ui`. If your app is not ready yet, complete the [shadcn installation guide](https://ui.shadcn.com/docs/installation) first.

Run the following commands from the root of that prepared app:

```bash
# Install one component
npx shadcn@latest add https://cambridgemonorail.github.io/WallRun/registry/registry.json auto-paging-list

# Install several at once
npx shadcn@latest add https://cambridgemonorail.github.io/WallRun/registry/registry.json metric-card event-card schedule-gate
```

This uses the shadcn CLI to copy the selected components into your codebase along with the files and npm dependencies they need. Once installed, you own that code and can adapt it to your project.

Choose this route when you want WallRun signage primitives inside an existing application. Clone the full repository only if you also want the demo site, player scaffolding, deployment scripts, or the rest of the workspace tooling.

## Start With The Right Surface

Use `SignageContainer` in real application screens.

Use `ScreenFrame` in stories, previews, and QA when you need to simulate an actual display resolution or safe area.

That distinction matters:

- `SignageContainer` is the app-facing full-screen shell.
- `ScreenFrame` is the preview and validation shell.

## Build In This Order

1. Establish the surface with `SignageContainer`.
2. Add structure with `SignageHeader` or `SplitScreen`.
3. Fill zones with blocks and primitives.
4. Wrap unstable regions with behaviours such as `ContentRotator`, `ScheduleGate`, or `OfflineFallback`.

That order keeps the screen stable and makes operational logic easier to reason about.

If you already know the screen shape but need to make it fit a venue or customer brand, read [Theming](?path=/docs/theming--documentation) before you start adding ad hoc class overrides to stories or pages.

## Minimal Realistic Example

```tsx
import {
  ContentRotator,
  FullscreenHero,
  OfflineFallback,
  SignageContainer,
  SignageHeader,
} from '@wallrun/shadcnui-signage';

const fallback = (
  <FullscreenHero
    title="Live notices unavailable"
    subtitle="Please check reception for updates"
    variant="dark"
  />
);

export function LobbyScreen() {
  return (
    <SignageContainer variant="blue">
      <div className="flex h-full flex-col gap-10 px-12 py-14 lg:px-16 lg:py-16">
        <SignageHeader
          tag="Main lobby"
          title="Welcome to WallRun"
          subtitle="Arrivals, notices, and live building updates"
        />

        <OfflineFallback isOnline isHealthy fallback={fallback}>
          <ContentRotator intervalMs={6000} transition="fade">
            <FullscreenHero
              title="Check in at reception"
              subtitle="Visitor badges available at desk 2"
              variant="dark"
            />
            <FullscreenHero
              title="Today at 14:30"
              subtitle="Product demo in Studio A"
              variant="dark"
            />
          </ContentRotator>
        </OfflineFallback>
      </div>
    </SignageContainer>
  );
}
```

## Common First Routes Through The Library

Choose a route based on the kind of screen you are building:

### Welcome Or Arrival Screens

- `Signage/Layouts/SignageContainer`
- `Signage/Layouts/SignageHeader`
- `Signage/Blocks/FullscreenHero`
- `Signage/Behaviour/ContentRotator`

### Menu Boards

- `Signage/Blocks/MenuBoard`
- `Signage/Primitives/MenuSection`
- `Signage/Primitives/MenuItem`
- `Signage/Behaviour/ScheduleGate`

### Lobby, Schedule, And Operations Screens

- `Signage/Primitives/SignagePanel`
- `Signage/Primitives/MeetingRow`
- `Signage/Primitives/InfoList`
- `Signage/Behaviour/OfflineFallback`
- `Signage/Behaviour/StaleDataIndicator`

## What To Look For In Storybook

The best stories here do more than render a component. They should show:

- a plausible screen context
- readable type hierarchy at display scale
- a clear reason the component exists
- operational behaviour that matches real unattended screens

If a story reads like a small component test, treat it as API reference, not design guidance.
