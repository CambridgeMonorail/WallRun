# Best Practices for Digital Signage

This guide reflects the way the signage docs are structured in Storybook: establish a stable screen shell first, then add content blocks, then add behaviour around those blocks.

## Build Screens In Layers

The most reliable signage screens are built in four steps:

1. Define the display surface.
2. Define stable layout regions.
3. Fill those regions with readable content.
4. Add timing, paging, gating, or fallback behaviour around whole regions.

If behaviour shows up before the screen has a clear structure, the result usually reads like a component demo rather than a finished display.

## 1. Start With A Stable Surface

Use `SignageContainer` for real application screens.

Use `ScreenFrame` for previews, Storybook, and QA when you need to simulate the actual target resolution and safe area.

```tsx
<ScreenFrame resolution="1080p" scale={0.5}>
  <SignageContainer variant="blue">...</SignageContainer>
</ScreenFrame>
```

Why this matters:

- the screen aspect ratio is known in advance
- the safe area can be checked before deployment
- the rest of the composition has a predictable canvas

## 2. Establish Regions Before Behaviour

Use layouts such as `SignageHeader`, `SplitScreen`, and fixed panels to decide where content belongs before you add operational logic.

```tsx
<SignageContainer variant="teal">
  <div className="flex h-full flex-col gap-10 px-12 py-14">
    <SignageHeader
      title="Visitor arrivals"
      subtitle="Check-in, notices, and live updates"
    />

    <SplitScreen ratio="60-40">
      <MainZone />
      <SidebarZone />
    </SplitScreen>
  </div>
</SignageContainer>
```

The shell should still make sense even if no behaviour is active yet.

## Theme At The Right Layer

Branding decisions should follow the same layering discipline as layout decisions.

Prefer this order:

1. Override shared theme tokens in the consuming app when the whole signage surface should shift.
2. Use component-level class hooks when one screen needs stronger emphasis than the rest of the app.
3. Avoid one-off story-level overrides unless the page is deliberately demonstrating customization.

The dedicated `Theming` page explains the current split between shared CSS variables in `common-tailwind` and screen-specific styling hooks in `shadcnui-signage`.

## 3. Favour Whole Message States Over Dense Fragments

The audience is standing across the room, not scanning a laptop screen.

Prefer:

- one dominant headline
- a small number of supporting facts
- clear region boundaries
- large consistent type sizes

Avoid:

- too many competing highlights
- tiny labels spread across the whole screen
- layouts that depend on unpredictable text length
- mixing unrelated content modes in one panel

## 4. Add Behaviour Around Regions, Not Inside Fragments

Use behaviours to swap, page, or protect complete content states.

Good pattern:

```tsx
<OfflineFallback fallback={<FallbackPanel />}>
  <ContentRotator intervalMs={6000} transition="fade">
    <WelcomeState />
    <EventState />
    <WeatherState />
  </ContentRotator>
</OfflineFallback>
```

Weak pattern:

- one behaviour wrapping a tiny label
- another behaviour wrapping one row inside the same card
- several timing rules competing inside a region with no clear state boundary

The screen should have obvious state ownership.

## Readability Rules

### Distance First

- body text should usually stay large enough to read from several metres away
- headings should dominate the screen clearly
- line length and line count should stay controlled

### Clamp Dynamic Content

Use clamping and fixed-height regions so the layout does not collapse when data changes.

```tsx
<div className="h-[400px] overflow-hidden">
  <h1 className="line-clamp-2">{dynamicTitle}</h1>
</div>
```

### Keep Contrast Obvious

- use strong foreground/background contrast
- reserve accent colours for emphasis and status
- avoid mid-tone-on-mid-tone combinations

## Behaviour-Specific Guidance

### `ContentRotator`

Rotate full content states, not minor fragments. The viewer should understand each slide as a distinct screen state.

### `AutoPagingList`

Use paging when the content is sequential and too long for one screen, but still deserves large readable rows. If the content feels cramped even after paging, the problem is density, not pagination.

### `ScheduleGate`

Gate complete daypart or mode-specific regions. Avoid scattering schedule logic across individual labels or rows.

### `OfflineFallback`

Fail into a deliberate replacement state, not a blank panel or apology message. The fallback should still help the viewer act.

## Common Failure Modes

### Responsive Website Thinking

Avoid treating signage like a generic responsive page with many breakpoints and fluid layout behaviour. Most signage screens target known resolutions and should behave deterministically.

### Auto-Sized Regions

Avoid layouts that grow and shrink based on feed content. Fixed or carefully bounded regions are easier to validate and safer to run unattended.

### Behaviour Without A Shell

If the behaviour demo is the only interesting part of the screen, the screen composition is probably underdesigned. The shell should remain coherent before rotation, paging, or fallback logic is added.

## Validation Checklist

Before you trust a signage screen, check:

- does the screen still read clearly on first render?
- does long data stay within its region?
- does the layout survive content loss or feed mutation?
- does each timed or gated state feel intentional?
- does the screen still make sense from distance?

That is the standard the Storybook signage examples should meet as well.
