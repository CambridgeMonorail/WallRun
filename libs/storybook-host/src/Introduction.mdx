# WallRun Storybook

WallRun is a component library and demo surface for building digital signage as software. The Storybook is most useful when you treat it as a screen-composition guide, not a catalogue of isolated widgets.

The primary library is `@wallrun/shadcnui-signage`. It is built for full-screen layouts, known resolutions, distance readability, and always-on operation on hardware such as BrightSign players.

You can use this Storybook in two ways:

- as composition guidance when you are working inside the WallRun workspace
- as installation guidance when you want to add specific signage components to an existing React app through the published shadcn registry

## Install From The Published Registry

If you do not need the full monorepo, you can install individual signage components from the published registry.

Before running these commands, make sure you are inside a React app that is already set up for Tailwind CSS and `shadcn/ui`. If your app is not ready yet, follow the [shadcn installation guide](https://ui.shadcn.com/docs/installation) first.

Run the following commands from the root of that prepared app:

```bash
# Install one component
npx shadcn@latest add https://cambridgemonorail.github.io/WallRun/registry/registry.json clock

# Install several components at once
npx shadcn@latest add https://cambridgemonorail.github.io/WallRun/registry/registry.json metric-card event-card schedule-gate
```

These components are copied into your application codebase. They are not installed as a locked runtime package, so you can edit them freely once added.

## What This Storybook Is For

Use this Storybook to answer three practical questions:

- Which components establish the screen surface and layout skeleton?
- Which components carry the content hierarchy for a given signage use case?
- Which behaviours keep a screen stable when content rotates, data goes stale, or time windows change?

If a page only proves that a prop exists, it is not enough. The stronger examples show how a real screen is assembled and why the component belongs in that composition.

## How The Signage Library Fits Together

The `Signage` section is organised in the same order you would usually build a screen:

### 1. Primitives

Use primitives to establish readable content units and preview constraints.

- `ScreenFrame` previews a target display resolution and safe area.
- `DirectoryCard`, `MeetingRow`, `InfoList`, and related primitives carry the actual information hierarchy.

### 2. Layouts

Use layouts to define stable regions before adding content.

- `SignageContainer` creates the full-screen surface.
- `SignageHeader` establishes page-level framing and hierarchy.
- `SplitScreen` divides a screen into predictable zones.

### 3. Blocks

Use blocks when a common signage pattern deserves a higher-level composition.

- `FullscreenHero` for welcome, event, and alert states.
- `MenuBoard` for menu and daypart screens.
- `InfoCardGrid` for structured promotional or dashboard-style panels.

### 4. Behaviour

Add behaviour only after the screen has a stable content structure.

- `ContentRotator` loops through whole content states.
- `Countdown` turns a deadline into the dominant message.
- `OfflineFallback`, `ScheduleGate`, `AutoPagingList`, and `StaleDataIndicator` keep unattended screens operational.

## Recommended Reading Order

If you are new to the signage library, read the docs in this order:

1. `Introduction`
2. `Getting Started`
3. `Theming`
4. `Signage/Layouts/SignageContainer`
5. `Signage/Blocks/FullscreenHero` or `Signage/Blocks/MenuBoard`
6. `Signage/Behaviour/ContentRotator` and `Signage/Behaviour/OfflineFallback`

That sequence moves from screen shell, to theme ownership, to composition, to operational behaviour.

## A Typical Screen Assembly

```tsx
import {
  ContentRotator,
  FullscreenHero,
  OfflineFallback,
  SignageContainer,
  SignageHeader,
} from '@wallrun/shadcnui-signage';

export function LobbyWelcomeScreen() {
  return (
    <SignageContainer variant="teal">
      <div className="flex h-full flex-col gap-10 px-12 py-14 text-white lg:px-16 lg:py-16">
        <SignageHeader
          tag="Main lobby"
          title="Visitor arrivals"
          subtitle="Welcome, queue guidance, and live building notices"
        />

        <OfflineFallback
          isOnline
          isHealthy
          fallback={<FullscreenHero title="Information unavailable" />}
        >
          <ContentRotator intervalMs={6000} transition="fade">
            <FullscreenHero
              title="Welcome to WallRun"
              subtitle="Check in at reception"
            />
            <FullscreenHero
              title="Today at 14:30"
              subtitle="Product demo in Studio A"
            />
          </ContentRotator>
        </OfflineFallback>
      </div>
    </SignageContainer>
  );
}
```

The pattern is deliberate:

1. Establish the screen surface.
2. Define the main hierarchy.
3. Add content blocks.
4. Add behaviours around those blocks, not inside random fragments.

## Related Libraries

The other libraries in this Storybook support the broader WallRun workspace:

- `@wallrun/shadcnui` provides the base UI primitives.
- `@wallrun/shadcnui-blocks` provides more general composed web components.
- `@wallrun/landing` and `@wallrun/shell` support the demo website chrome rather than signage content itself.

Start in `Signage` unless you are working on the website rather than the display content.
